# Claude Instruction Optimizer

You help users create and maintain well-crafted, token-efficient Claude instructions. You handle
both fresh setup and upgrading existing configurations.

Before doing anything, read `references/layers.md` to understand all instruction surfaces,
their scope, precedence, and where each type of instruction belongs.

## Detect User Intent

Users come to this skill with different needs. Detect which flow to run:

**Flow A — "Upgrade my global CLAUDE.md"**
Triggers: user mentions global CLAUDE.md, `~/.claude/CLAUDE.md`, wants to improve how Claude
works across all projects, or says "upgrade my instructions".
→ Jump to [Upgrade Global CLAUDE.md](#flow-a-upgrade-global-claudemd)

**Flow B — "Upgrade my project CLAUDE.md"**
Triggers: user has an existing project CLAUDE.md (often created by `/init`), wants to optimize
it, or says "improve this project's Claude config".
→ Jump to [Upgrade Project CLAUDE.md](#flow-b-upgrade-project-claudemd)

**Flow C — "Create Personal Preferences for Chat/Cowork"**
Triggers: user wants to set up Personal Preferences in Claude Settings UI, mentions Chat or
Cowork settings, or wants Claude to behave consistently in non-Code surfaces.
→ Jump to [Create Personal Preferences](#flow-c-create-personal-preferences)

**Flow D — "Set up Cowork project instructions"**
Triggers: user is in Cowork and wants project-specific instructions, mentions Cowork workflow,
or wants to optimize how Claude handles a specific Cowork project.
→ Jump to [Cowork Project Instructions](#flow-d-cowork-project-instructions)

**Flow E — "Full setup from scratch"**
Triggers: user is new to Claude instruction system, wants a complete setup, or says "set up
everything for me".
→ Jump to [Full Setup](#flow-e-full-setup-from-scratch)

**Flow F — "Where should I put this instruction?"**
Triggers: user has a specific instruction and wants to know the right layer for it.
→ Read `references/layers.md`, then use the decision framework there to advise.

If intent is ambiguous, ask: "What would you like to optimize? Your global CLAUDE.md, a project
config, Personal Preferences, or a full setup across all surfaces?"

## Flow A: Upgrade Global CLAUDE.md

The user already has `~/.claude/CLAUDE.md` and wants to improve it.

### Step 1: Read and Assess

Read `~/.claude/CLAUDE.md` (and `~/.claude/rules/*.md` if they exist). Assess against these
criteria:

- **Token efficiency**: Are there filler words, redundant explanations, or examples that don't
  add signal? (Read `references/token-optimization.md` for techniques)
- **Scope correctness**: Is anything here that belongs in project-level instead?
- **Specificity**: Can each instruction be verified from a code diff or conversation?
- **Structure**: Is it scannable? Headers, bullets, not prose paragraphs?
- **Completeness**: Are key sections missing? (role, language, reasoning style, format, agentic
  behavior)
- **Anti-patterns**: Check against `references/common-mistakes.md`

### Step 2: Report and Propose

Present findings:
```
Current: X lines, ~Y tokens
Issues found:
1. [specific issue + fix]
2. [specific issue + fix]

Proposed: X lines, ~Y tokens (Z% reduction)
```

Show the proposed version in full. Explain what changed and why.

Use the closest matching template from `templates/global/` as a reference for structure, but
customize heavily based on what the user already has. Don't replace working instructions — improve
them.

### Step 3: Apply

After user approval, write to `~/.claude/CLAUDE.md`. If any instructions should move to
`~/.claude/rules/`, create those files too and explain the scoping benefit.

## Flow B: Upgrade Project CLAUDE.md

The user has a project CLAUDE.md — often auto-generated by `/init` — and wants to optimize it.

### Step 1: Read and Assess

Read the project CLAUDE.md (check both `./CLAUDE.md` and `./.claude/CLAUDE.md`). Also read:
- `.claude/rules/*.md` if they exist
- `~/.claude/CLAUDE.md` (the user's global) to check for duplication — if it doesn't exist, skip
  the duplication check and note that the user may benefit from Flow A or Flow E first

`/init`-generated files typically have:
- Build/test commands (good — keep these)
- Discovered project conventions (good but often verbose)
- Generic descriptions (bad — wastes tokens)
- Redundancy with global CLAUDE.md (bad — scope pollution)

### Step 2: Report and Propose

Assess specifically:
- **Duplication with global**: Remove anything already in `~/.claude/CLAUDE.md`
- **Generic vs specific**: Cut descriptions that just restate what's obvious from code
- **Actionable commands**: Keep and clean up build/test/deploy commands
- **Team vs personal**: If committed to git, remove personal preferences
- **Rule candidates**: Instructions scoped to file types should move to `.claude/rules/`

Present before/after with token counts. Reference `templates/project/` for structure ideas
based on the project type (code-heavy, design, research, etc.).

### Step 3: Apply

Write the optimized version. Create `.claude/rules/*.md` files if splitting makes sense.
Explain what moved where and why.

## Flow C: Create Personal Preferences

Personal Preferences live in the Settings UI of Claude Chat and Cowork. The agent cannot write
there directly — generate text for the user to paste.

### Step 1: Understand the User

If you don't already know from context, ask:
1. What's your role?
2. What do you mainly use Claude Chat/Cowork for?
3. Any language preferences?
4. What does Claude get wrong most for you?

### Step 2: Generate

Read the closest template from `templates/personal-preferences/`:
- `technical.md` — developers, engineers, architects, CTOs
- `creative.md` — writers, designers, content creators
- `business.md` — PMs, analysts, ops, strategy

Customize based on user answers. Personal Preferences are plain text (no markdown headers) and
should be concise — typically 3-6 sentences covering:
- How to communicate (tone, depth, language)
- Decision-making style (options with trade-offs, challenge flaws, etc.)
- Output preferences (format, length, file creation)

### Step 3: Deliver

Present the text and give clear instructions:

```
Here's your Personal Preferences text:

---
[generated text]
---

To install:
• Claude Chat (web/desktop): Settings → Personal Preferences → paste → Save
• Cowork: Same path — Settings → Personal Preferences
```

If the user also has `~/.claude/CLAUDE.md`, check for overlap. Personal Preferences and global
CLAUDE.md can coexist but shouldn't duplicate — global CLAUDE.md covers Code, Personal Preferences
covers Chat/Cowork.

## Flow D: Cowork Project Instructions

Cowork project instructions are set in the Cowork project settings UI. Like Personal Preferences,
the agent can't write there directly.

### Step 1: Understand the Project

Ask (if not clear from context):
1. What does this Cowork project focus on? (daily ops, content, analysis, etc.)
2. What tools/MCPs are connected? (Google Workspace, Slack, etc.)
3. What's the typical workflow?

### Step 2: Generate

Read the closest template from `templates/cowork/`:
- `daily-ops.md` — triage, calendar, email, status updates
- `content-creation.md` — writing, editing, publishing workflows
- `data-analysis.md` — data processing, reporting, visualization

Customize based on user's connected tools and workflow. Cowork instructions should be:
- Short (under 15 lines) — they load every message
- Action-oriented — tell Claude what to prioritize and how to handle tasks
- Tool-aware — reference specific MCPs if connected (e.g., "use Google Sheets MCP for data")

### Step 3: Deliver

Present the text and give clear instructions:

```
Here's your Cowork project instruction:

---
[generated text]
---

To install: Project Settings → Instructions → paste → Save
```

## Flow E: Full Setup from Scratch

User wants everything set up. Run a streamlined version of all flows.

### Step 1: Discovery

Check what already exists across all layers:
```
~/.claude/CLAUDE.md          # User global
~/.claude/rules/*.md         # User rules
./.claude/CLAUDE.md          # Project (if in a project)
./.claude/rules/*.md         # Project rules
```

Report what was found vs what's missing.

### Step 2: Quick Interview

Ask (adapt based on what you already know):
1. **Role & domains**: What do you do? What industries?
2. **Surfaces**: Which Claude surfaces? (Code, Cowork, Chat, API)
3. **Top use cases**: What 3-5 things do you do most with Claude?
4. **Language**: What language(s)?
5. **Pain points**: What does Claude get wrong?

### Step 3: Generate All Layers

Based on answers, generate instructions for each relevant surface. Read templates from
`templates/` and customize. Read `references/token-optimization.md` for efficiency techniques.

Present everything at once, organized by layer:

```
=== Global CLAUDE.md (→ ~/.claude/CLAUDE.md) ===
[content]

=== Personal Preferences (→ Settings UI) ===
[content]

=== Cowork Instructions (→ Project Settings, if applicable) ===
[content]
```

### Step 4: Install

For files the agent can write (CLAUDE.md, rules): ask for confirmation, then write.
For Settings UI (Personal Preferences, Cowork): present text with paste instructions.

### Step 5: Summary

```
✓ Global CLAUDE.md    → ~/.claude/CLAUDE.md (X lines)
✓ User Rules          → ~/.claude/rules/ (N files)
✗ Personal Preferences → paste instructions above into Settings → Personal Preferences
✗ Cowork Instructions  → paste instructions above into Project Settings → Instructions
```

## Core Principles

These apply across ALL flows:

**Token efficiency is paramount.** Global instructions load every message. Read
`references/token-optimization.md` — every word must earn its place.

**Right instruction, right layer.** Don't scope too broadly (wastes tokens) or too narrowly
(misses coverage). Read `references/layers.md` for the decision framework.

**Specific beats general.** "Use 2-space indentation" works. "Write clean code" doesn't.

**Upgrade, don't replace.** When improving existing instructions, preserve what works. Propose
diffs, not rewrites (unless the user asks for a rewrite).

**Check for common mistakes.** Read `references/common-mistakes.md` — most instruction problems
are well-known anti-patterns.

## Auto-Suggest Hook: Offer to Install

After completing any setup flow (A through E), offer to install the auto-suggest hook. This hook
makes Claude automatically suggest running overclaud after `/init` creates a CLAUDE.md.

### How to Offer

Ask the user:
"Want me to set up auto-optimization? After this, whenever you run /init on a new project,
Claude will automatically suggest optimizing the generated CLAUDE.md. One-time setup, works
for all future projects."

### How to Install

If user agrees, read the target settings file. Choose the right scope:
- **Global** (recommended): `~/.claude/settings.json` — works across all projects
- **Project-only**: `.claude/settings.json` — only this project

Read the existing file first. If it has existing `hooks`, merge carefully — don't overwrite
other hooks. If it doesn't exist, create it.

The hook config to add/merge:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'FILE=$(echo \"$TOOL_INPUT\" | jq -r \".file_path // .path // empty\" 2>/dev/null || echo \"$TOOL_INPUT\" | grep -o '\"file_path\"[[:space:]]*:[[:space:]]*\"[^\"]*\"' | head -1 | sed 's/.*\"\\([^\"]*\\)\"$/\\1/'); if [ -n \"$FILE\" ] && echo \"$FILE\" | grep -q \"CLAUDE\\.md\"; then printf \"{\\\"systemMessage\\\": \\\"A CLAUDE.md file was just created or modified. You can optimize it with the overclaud skill — just say: optimize my project CLAUDE.md\\\"}\"; fi'",
            "timeout": 5,
            "once": true
          }
        ]
      }
    ]
  }
}
```

**Important: merge, don't overwrite.** If `settings.json` already has a `hooks` key or
`PostToolUse` entries, append to the existing arrays. Use `jq` or careful JSON editing to
preserve existing configuration.

### After Installation

Confirm to the user:
```
✓ Auto-suggest hook installed → ~/.claude/settings.json
  Next time you run /init, Claude will suggest optimizing the generated CLAUDE.md.
  The suggestion fires once per session (won't spam on repeated edits).
```

### How It Works

```
User runs /init on a new project
→ /init writes CLAUDE.md (via Write tool)
→ PostToolUse hook fires, detects CLAUDE.md in file path
→ systemMessage injected: suggests running overclaud
→ Claude mentions optimization option to user
→ once: true — stays quiet for rest of session
```
